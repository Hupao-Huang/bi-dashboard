## 一句话总览

这是**松鲜鲜公司内部用的 BI 数据看板系统**。它把原本散落在各个系统里的数据——吉客云 ERP（进销存）、用友 YonBIP（采购/库存）、合思（费控报销）、还有 11 个电商平台每天用 RPA 机器人导出的 Excel——**全部汇总到一个数据库里**，再做成一屏屏的销售、库存、采购、财务、物流、客服分析看板，给公司 **35～43 位内部同事**（管理层 + 各业务部门）看数用。

**怎么搭的（一句话技术版）**：

- **前端**（用户看到的网页）：React + 蚂蚁 AntD 组件 + ECharts 图表，十几个业务板块、约 30 个页面。
- **后端**（在后台取数算账的）：Go 语言写的一个程序 `bi-server.exe`，跑在内网 8080 端口，直连 MySQL 数据库。约 120 个接口文件。
- **数据搬运工**：约 60 个独立小工具（`import-*` 导 Excel、`sync-*` 同步外部系统），由 Windows 计划任务（schtasks）定时在后台跑。
- **部署**：装在一台 Windows 11 机器上，公司内网访问，单人维护。

**规模一览**：

| 部分 | 规模 |
|------|------|
| 后端接口（handler） | ~120 个文件，约 4.2 万行 |
| 定时/导入小工具（cmd） | ~60 个独立程序 |
| 前端业务页面 | 十几个板块、约 30 个页面 |
| 数据库 | 按月分表 + 11 平台运营表 + 物化预聚合表（数百张） |
| 外部数据源 | 吉客云 / 用友 YS / 合思 / 钉钉 / RPA Excel |

## 后端是怎么跑起来的（启动 → 鉴权 → 路由 → 处理 → 数据库）

整个后端就是**一个 Go 程序**（编译出来叫 `bi-server.exe`），跑在公司内网的 8080 端口上。它没有用任何"网页框架"，而是直接用 Go 自带的标准网络库（`net/http`）手搓的。好处是简单、依赖少、好排查；代价是路由这些都得自己一条条写。下面按"一个请求从进门到拿到数据"的顺序讲。

### 1. 开机启动做了哪些事（入口文件 `server/cmd/server/main.go`，约 276 行）

程序一启动，按顺序把"地基"打好：

| 步骤 | 干了啥（大白话） |
|------|------------------|
| 读配置 | 从 `config.json` 读数据库密码、各家外部系统的钥匙（吉客云/用友/合思/钉钉/影刀/AI）。**关键一道保险**：如果 webhook 密钥没填，直接拒绝启动，防止同步接口被外人乱调 |
| 连数据库 | 连上 MySQL，开一个"连接池"（最多 100 条线同时连），连不上就停机报错 |
| 建账号表 | 自动检查并补齐用户、角色、权限这些系统表（`EnsureAuthSchemaAndSeed`），保证管理员账号在 |
| 备好外部系统的"接线员" | 钉钉通知、影刀 RPA、AI 助手、用友 YS 这几个客户端各自初始化好；哪个没配钥匙就自动关掉那块功能，不影响其它 |
| 开几个后台值班程序 | 定时任务健康巡检（卡死/失败自动钉钉告警）、影刀任务状态巡查、过期数据清理、综合看板后台预热（让首页秒开）、合思审批排队员等。这些都在后台默默跑，不挡前台请求 |
| 开门接客 | 在 8080 端口启动，开始接收前端请求 |

### 2. 每个请求都套了几层"安检"（中间件）

前端发来的每个请求，进门后会**层层包裹**地过安检，顺序是：

1. **访问日志** —— 记下谁、什么时间、访问哪个地址、花了多少毫秒、成功还是失败（出错自动标红）。
2. **崩溃兜底（Sentry）** —— 万一某段代码崩了，不会让整个服务挂掉，而是返回"服务器繁忙"并把错误推给监控，管理员能收到告警。
3. **跨域 + 限流（CORS）** —— 只允许公司自己几个网址来访问；上传文件最大 10MB、其它请求最大 2MB，防止有人发超大包把服务撑爆。
4. **登录校验（`RequireAuth`）** —— 没登录的一律挡回去。
5. **权限校验（`RequirePermission` 等）** —— 不光要登录，还得有"看这个页面的权限"。代码里区分了三种粒度：要某一个权限、要多个里任意一个、要多个全有。比如看综合看板需要"综合看板查看"权限，财务导出还得同时有"财务查看 + 数据导出"两个权限。

这套安检是用"包装函数"实现的（`protected`、`pageProtected` 这些），每条路由注册时挑合适的包装层套上去。

### 3. 网址到处理逻辑的对应关系（路由）

所有对应关系都**集中写在 `main.go` 一个大列表里**，一条 `mux.HandleFunc("网址", 处理函数)` 就是一条对应规则，类似一本"电话簿"：哪个网址打过来，转给哪个处理员（handler）接。全部网址都以 `/api/` 开头，比如：

- `/api/overview` → 综合看板数据
- `/api/department` → 各部门明细
- `/api/finance/report` → 财务报表
- `/api/ai-assistant/ask` → AI 智能助手问答
- `/api/webhook/sync-ops` → 给 RPA 同步用的对外接口

还有个省钱机制叫**缓存**：像昨天/上月这种"永远不会变"的数据，结果会缓存起来（T+1 数据缓存 24 小时、合思费控因为每 15 分钟同步一次就缓存 15 分钟），下次同样的请求直接给缓存，不用再去数据库慢慢算。

### 4. 处理员（handler）干活 + 统一的回复格式

接活的处理函数都集中在 `internal/handler/` 这个大目录里（最大的一块，约 125 个文件）。它们做的事就是：查 MySQL、按业务规则算一算（KPI、率、调拨等），然后返回。

**所有回复都是同一个固定格式**（`writeJSON`，在 `dashboard.go`）：
```
{ "code": 200, "data": 真正的数据 }
```
所以前端拿数据永远是 `结果.data.xxx`，不是直接 `结果.xxx`。出错时走 `writeError`，格式是 `{code, error}`。这个统一约定让前后端配合不容易出岔子。

### 用了什么技术栈（一句话）

**Go 语言 + 自带的 `net/http` 标准库（没有用任何 web 框架）+ MySQL 数据库**。额外搭了：Sentry（崩溃监控）、slog（结构化 JSON 日志，方便日志系统检索）。部署在 Windows 11 上，定时任务用 Windows 自带的计划任务（schtasks），不是 Linux 的 cron。

## 数据从哪来、怎么进看板、库里存了啥

### 一句话总览
BI 看板自己不直接连任何外部系统。它背后有一个 MySQL 数据库（`bi_dashboard`），一堆"搬运工"程序定时把外部系统的数据搬进这个库，看板只负责从库里读数、算指标、画图。所以**外部系统 → 搬运工 → 数据库 → 看板**这条链路是整个项目的骨架。

### 一、数据从哪来（5 个外部系统）

| 外部系统 | 接进来的是什么数据 | 怎么接 | 关键代码在哪 |
|---|---|---|---|
| **吉客云 ERP**（销售/库存核心） | 销售单（已发货订单明细）、按商品的销售汇总、库存出入库流水（含成本价）、渠道清单 | 用账号密钥签名调它的 API，分页一批批拉 | `server/internal/jackyun/`（trade=销售单、sales_summary=销售汇总、stock_io=出入库、channel=渠道） |
| **YS 用友 YonBIP**（供应链） | 采购订单、委外加工订单、材料出库单、当前库存（现存量） | 调它的 4 个开放接口，**每秒最多 1 次**（限流），登录令牌 2 小时换一次 | `server/internal/yonsuite/` |
| **合思费控**（报销/费用） | 费用报销单据、单据详情、单据附件 | 调合思（易快报）开放接口 | `server/internal/handler/hesi.go` |
| **钉钉**（账号+通知，不是业务数据） | ①员工扫码登录看板 ②看板有事时主动给员工推消息 | OAuth 登录 + 机器人推送 | 登录：`auth_dingtalk.go`；推送：`server/internal/dingtalk/notify.go` |
| **RPA Excel（共享盘）** | 11 个电商平台（天猫、京东、拼多多、唯品会、抖音、飞瓜等）的店铺/商品/活动/客服等运营报表 | 机器人每天把 Excel 丢到共享盘（`Z:\` 或网络路径 `\\172.16.100.10\松鲜鲜资料库`），程序去读这些 Excel | 读盘：`server/internal/importutil/`；解析入库：`server/cmd/import-*`（14 个，每平台一个） |

> 补充：财务每月出的"考核利润表"和"业务预决算表"也是 Excel，分别由 `internal/finance/` 和 `internal/business/` 解析入库。另有"影刀 RPA"（`internal/yingdao/`），看板可以远程**启动/查看**影刀机器人去抓数据——它是个遥控器，本身不是数据。

要点：
- **两类搬运工**：`sync-*` 拉 API 数据（吉客云/YS/合思），`import-*` 读 Excel（电商平台 RPA）。它们都是独立的小程序，由 Windows 定时任务（schtasks，名字都带 `BI-` 前缀）按点触发。
- **物化预聚合**：明细表太大（每月约 200 万行），直接查会慢。所以每天凌晨 03:30 自动把明细"压缩"成现成的汇总表，物流仓储看板因此能从 7 秒提速到毫秒级。看板会自动判断有没有这张压缩表，没有就退回去查原始明细。

### 三、数据库里大概存了哪些东西（表分组 ↔ 存什么）

| 表分组 | 存的是啥业务数据 |
|---|---|
| **按月分表 `trade_YYYYMM` 系列**（`trade_` 销售单 / `trade_goods_` 商品明细 / `trade_package_` 包裹 / `stock_snapshot_` 库存快照） | 吉客云的**已发货订单**，按发货月份拆成一个月一张表（每月约 200 万行，避免单表太大）。这是销售额、销量、客单价、物流仓储分析的底料。 |
| **运营 RPA `op_*_daily` 系列（约 70 张表）** | 11 个电商平台的每日运营数据：店铺概况、商品、推广活动、客服、会员、人群等。比如 `op_tmall_*`（天猫）、`op_jd_*`（京东）、`op_pdd_*`（拼多多）、`op_vip_*`（唯品会）、`op_tmall_cs_*`（天猫客服/淘客）。 |
| **物化预聚合表** `warehouse_flow_summary`、`sales_goods_summary_monthly` | 由上面的明细表预先算好的"成品汇总账"，专门给看板做到秒开。源数据一变就重新生成。 |
| **YS 用友 `ys_*` 表** `ys_purchase_orders` / `ys_subcontract_orders` / `ys_material_out` / `ys_stock` | 供应链数据：采购订单、委外加工订单、材料出库、当前库存。支撑采购计划、库存预警、供应链看板。 |
| **财务 `finance_report` / `business_budget_report` / `hesi_*`** | `finance_report`=考核利润表（按部门）；`business_budget_report`=业务预决算（每月快照、12 个月预算 vs 实际）；`hesi_*`=合思费用报销单据/明细/附件。 |
| **系统表** `users` / `roles` / `notices` / `audit_log` / `sales_channel` | 跟业务无关的"系统底座"：用户、角色权限、版本公告、操作审计日志、销售渠道字典。 |

> 看板的所有页面（综合、电商、社媒、线下、分销、客服、财务、供应链等）本质上都是从上面这几组表里取数再算指标，外部系统宕机也不影响看板查历史数据——因为数据早已搬进了自己的库。

## 每天自动在后台跑的活（server/cmd 程序清单）

下面这些是看板背后“自己干活”的一堆小程序。每个文件夹是一个独立小程序，编译成一个单独的 `.exe`。它们靠 Windows 的“定时任务”（任务名都以 **BI-** 开头）按点自动跑，也可以在网页后台点“手动同步”按钮临时触发。数据来源就 4 个地方：**吉客云ERP**（订单/库存/汇总）、**用友YS**（采购/委外/包材库存）、**合思**（报销费控）、**RPA Excel**（机器人每天往 Z 盘下载的各电商平台运营报表）。

> 一句话记法：`import-*` = 搬 Excel 进库；`sync-*` = 和外部系统对接同步；物化/快照 = 把慢查询提前算好、把库存存历史；预测 = 算未来销量；探针/审计 = 平时不跑、出问题时手动查的小工具。

## 部署与运维要点（怎么启动、重启、发版、出事怎么救）

> 这一节是给维护的人看的"操作手册速查"，整理自项目内部约定（CLAUDE.md）。

### 怎么跑起来 / 重启

- **前端**：是"打包成静态文件再用 serve 托管"的模式。改了页面（`.tsx`）**必须先 `npm run build`** 才能让大家看到新版，开发用的热更新不算生产。
- **后端**：进 `server/` 目录编译 `go build -o bi-server.exe ./cmd/server`，把 exe 拷到 `server/` 根目录，然后**只杀 8080 端口对应的进程**再重启（不要一股脑杀所有 go/node 进程，会误伤别的服务）。

### 发版流程（"提交代码"和"对外发版"是两件事）

- **提交阶段**：每个小改动正常提交、上生产，但**不发公告**、不升版本号，不打扰用户。
- **发版阶段**：一组功能收尾时，4 步连发——① 升版本号（一组功能闭环升中间位，紧急修 bug 才升末位）② 写 CHANGELOG 段落 ③ 打 git 标签 ④ 往公告表插一条**面向用户口吻**的公告（不写技术变量名/文件名/具体业务数字）。

### 上线 4 条规矩（每次改动前自查）

1. **错峰重启**：避开每天 09:00–09:30、14:00–14:30，以及月初 1–5 号、月底盘点期。
2. **重大改动发钉钉**：新增页面 / 改业务规则 / 调字段时，提前在群里说一句"几点重启 30 秒，刷新即可"。
3. **业务红线必须先本地跑通再上**：业务规则与算法（KPI 公式、客服平均、财务科目、库存计费）、数据库加删改表、定时任务调度、登录与权限、路由——这几类要格外谨慎、最好二审。
4. **留好回滚路**：万一改坏，`git revert` 撤回 + 重新编译重启，3 分钟救场。
