## 一、背景与范围

### 1.1 现状 (2026-06-17 实查)

- 张俊 BI 看板 `user_id=44` (合思 staff_id 现场核, 不写死)。
- 文档点名两类模板, 实测对应两个 spec (2026-06-17 查库):
  - **付款单 (票到付款/票到核销)** = `ID01KgaO6dcZtR` (code **B 开头**), 全库 **4076 张**, 张俊当前审 **70 张**, **有发票** (填充 91~100%)。主战场。
  - **预付款单 (先款后票)** = `ID01FhdI9II9A3` (code **J 开头**), 全库 **1407 张**, 张俊当前审 **4 张**, **0 张有发票** (先款后票, 审批时发票还没到)。
- 张俊是付款单的当前审批人之一 (与黄宝洁/张勇/王嘉等轮流)。
- 张俊机器人此前已**彻底下线** (revert `84321d3`, handler 中已无"张俊"字样), 本次**从零重建**。
- 旧设计假设的基建"需新建", 现多数**已存在** (见 §5)。

### 1.2 单据范围 (跑哥拍板)

**做文档点名的两类模板: 付款单 (`ID01KgaO6dcZtR`, B 开头) + 预付款单 (`ID01FhdI9II9A3`, J 开头)。** 报销单 / 借款单 / 请购单 / 其他模板不在本次范围。

两类共用一套引擎, 但**按模板走不同规则子集** (见 §2.4): 付款单有发票走全套 A+B; 预付款单先款后票无发票, 跳过发票核验类规则。

### 1.3 输出模式 (跑哥拍板)

- **AI 建议给张俊看** (沿用 `suggestion` 字段路径), **不真审批** (不动 `hesi_approval_queue` / worker), dry-run。
- 误判容忍度高 (张俊看到可人工纠正)。
- 三档动作: `reject` 驳回 (能确定违规) / `manual` 转人工 (数据不全/拿不准, 不下结论) / `agree` 同意 (全部通过)。
- **安全底线: 数据缺失/拿不准一律转人工, 绝不硬驳。**

### 1.4 本次范围 = A 档 + B 档 (跑哥拍板 "A+B")

- **A 档「字段判定」(18 条)**: 靠单据自身字段 + 现成部门树/字典即可判。
- **B 档「发票核验」(4 条)**: 靠合思现成发票数据 (购买方/开票方/税额) 判, **无需外接 OCR**。
- **C 档「合同 OCR+LLM」(旧设计规则 24, ATT-02 合同关键字段抽取)**: **不在本次范围**, 后续单独成期。

### 1.5 关键事实: B 档无需外接 OCR (2026-06-17 实测)

合思发票数据已存于 `hesi_flow_invoice`, 含 OCR 出的关键字段, 实测填充率 (该模板 6628 行发票):

| 字段 | 列 | 填充率 |
|---|---|---|
| 购买方 | `buyer_name` | 94% (6234/6628) |
| 开票方/销售方 | `seller_name` | 91% (6016/6628) |
| 税额 | `tax_amount` | 100% (6628/6628) |
| 发票额 | `invoice_amount` / `total_amount` | 已有 |
| 验真 | `is_verified` | 已有 |

实测样本干净 (购买方=本方公司如"杭州世用食品科技", 开票方=供应商如"唐山市艺诚广告")。**未识别的 6~9% → 对应规则自动转人工。** 旧设计假设的 ¥0.05/张 OCR 服务 (百度/腾讯/阿里) **不再需要**。

> 预付款单 (`ID01FhdI9II9A3`) 实测 1407 张 **0 张有发票** (先款后票), 故 B 档发票核验对预付款单整组不适用 (§2.4)。

## 二、架构与触发

### 2.1 独立引擎

新建付款单审批引擎 (暂名 `AuditPayment`, 新文件 `hesi_audit_payment_rules.go`), 与樊雪娇报销引擎 `AuditDailyExpense` (`hesi_audit_rules.go`) **并排, 各判各的单据类型, 互不干扰**。

> 选型理由 (跑哥拍板"照搬樊雪娇打法"): 风险最低、复用现成 helper、张俊零学习成本; 两套规则分文件避免互相踩 (尤其 `hesi_audit_rules.go` 当前体量大且有在制改动)。否决: ① 塞进 `AuditDailyExpense` 加分流 (单函数混两套规则, 易踩); ② 通用配置化规则引擎 (过度设计, 规则磨合期, YAGNI)。

### 2.2 触发条件

```
当前审批人 = 张俊  AND  ( specification_id LIKE 'ID01KgaO6dcZtR%'   -- 付款单
                      OR specification_id LIKE 'ID01FhdI9II9A3%' ) -- 预付款单
```

在 `profile_hesi_pending.go` (`GetMyHesiPending`) 内, 仿樊雪娇触发块, 加 `isZhangJun && (付款单 spec 前缀 || 预付款单 spec 前缀)` 分支, 调 `AuditPayment` 而非 `AuditDailyExpense`。其他审批人/其他单据不进本引擎。引擎内按 spec 前缀判模板类型, 走对应规则子集 (§2.4)。

### 2.3 展示

张俊登录 BI → 自己待审列表每单带 AI 建议 (驳回/转人工/同意 + 理由), 详情弹窗内联标记; 与樊雪娇看到的同款 (复用 `HesiFlowDetailModal`)。

> 上线前实测张俊账号 (user_id=44) 能正常出建议; 其合思 staff_id 映射现场核 (旧 memory 值已变, 不写死)。备选: 张俊不常登则跑哥用管理员视角切换查看 (同樊雪娇)。

### 2.4 模板分支 (哪条规则对哪类模板生效)

| 规则组 | 付款单 (票到付款/核销, 有发票) | 预付款单 (先款后票, 无发票) |
|---|---|---|
| A 档非发票类 (A1 A2 A3 A5 A6 A7 A10 A11 A12 A13 A14 A17) | ✅ 全做 | ✅ 全做 |
| A8 明细行数 | ✅ B 开头: 无票费用可无发票, 其余必须有票 | ✅ J 开头: 可无明细 (放过) |
| A9 发票影像上传 | ✅ | ⏭️ 跳过 (无发票) |
| A4 / A15 / A16 核销类 | ✅ 仅"票到核销"子类型触发 | ⏭️ 跳过 (预付款单是原始预付, 非核销) |
| A18 支付金额上限 (依赖发票额) | ✅ | ⏭️ 跳过发票额部分 |

| B4 防重复付款 | ✅ | ✅ **重点做** (新文档点名预付款防重复) |

## 四、B 档规则清单 (发票核验, 4 条)

| # | 规则 | 判定逻辑 | 动作 | 数据源 |
|---|---|---|---|---|
| B1 | 收款方=开票方 | 收款方名 vs 发票 `seller_name`; 一单多张发票→匹配任一张即可 | **转人工** | `hesi_flow_invoice.seller_name` |
| B2 | 购买方=所属公司 | 发票 `buyer_name` vs 单据所属公司 (全部发票均须一致) | **驳回** | `hesi_flow_invoice.buyer_name` |
| B3 | 税额份数对账 | 发票 `tax_amount` 合计 + 张数 vs 单据填写的税额份数总计; **0 税正常放过** | **转人工** | `hesi_flow_invoice` 聚合 |
| B4 | 防重复付款 | 同收款方 + 金额完全一致的历史单 → 提示"疑似重复 + 原单号" (**付款单 + 预付款单都做**, 新文档点名预付款防重复) | **转人工** | `hesi_flow` 历史查询 |

## 六、错误处理与边界

1. **数据缺失 → 转人工, 绝不硬驳** (安全底线)。B 档发票字段未识别 (6~9%) 的单据, 对应规则转人工。
2. **一单多张发票**: B1 收款方匹配任一开票方即可; B2 要求全部发票购买方一致 (任一张≠所属公司→驳回)。
3. **0 税发票** (普票/免税): B3 不得据此误判, 0 税正常。
4. **raw_json 解析失败 / 关联单拉不到**: 该条规则降级转人工 + 留日志, 不影响其他规则。
5. **长 SQL 加 timeout** (B4 历史查询), 防锁链阻塞定时任务。
6. **金额比对精确到分** (A18 / B3), 用 decimal 不用 float。

## 十、业务红线与风险

- **业务规则改造, 每期实施前必须 `/code-review` 二审** (按 BI 看板 CLAUDE.md; 本机用 /code-review 不用 /codex)。

- 风险③审批人名单错 → 缓解: 名单实施时与张俊核实再写。
- 风险④历史防重误判 (合法二次付款) → 缓解: 仅转人工 + 提示原单号, 不驳回。
- **部署注意**: 改 `profile_hesi_pending.go` / 新 handler 文件后须重建 bi-server.exe + 错峰重启; 本文件与当前在制的"发票超期规则 (applyStableSubmitDate)"同处 handler 包, 提交时分清。
