# 销售日报(发货日报)设计文档

- 日期: 2026-07-01
- 位置: 供应链 →「销售日报」
- 来源需求: 跑哥给的 `销售日报.xlsx`(结构模板)+ `RPA映射表.xlsx`(渠道/箱规)+ `箱规拖规.xlsx`(箱规/托规)
- 一期口径: **发货日维度、只算 4 个仓、实时查(不建物化表)**

---

## 1. 背景与目标

业务现在每天手工拼一张《销售日报》——从 RPA/手工的源表(渠道单量/单品数量/货盘单量/箱规)里 VLOOKUP + SUMIFS 拼成三块矩阵。目标:把这张日报搬到 BI 看板,数据直接从吉客云发货订单库算,业务打开即看,不用再手工拼。

**这张日报的本质 = 发货量日报(不是销售额)**。核心指标:

- **发货单数**(报表叫「发货量」,源 sheet 叫「渠道单量」= 去重订单数)
- **发货单瓶数**(报表叫「发货件数」,备注"件数转换成单瓶数量")
- **单件比** = 单瓶 ÷ 单数(每单几瓶)
- **单均重量(kg)** = 重量 ÷ 单数
- **发货箱数** = 单瓶 ÷ 箱规;**发货托数** = 箱数 ÷ 托规
- **占比 / 环比**

三块:① 平台/渠道汇总 ② TOP10 单品 ③ TOP10 货品组合。每块都出 **当日 + 当月累计** 两组列。

## 2. 范围 / 非目标

### 一期做

- 供应链菜单新增「销售日报」页,版式照 `销售日报.xlsx` 三块。
- 数据实时从 `trade_*` / `trade_goods_*` 算,发货日维度,只算 4 个仓。
- 3 张映射(渠道 / 箱规 / 托规)落库 + 支持后续维护。
- 一个发货日选择器(默认最近有数的发货日)。

### 一期不做(YAGNI)

- **不建物化预聚合表**(方案②)。先实时查把口径/样式跟业务对齐,确认后再单独上物化提速。
- **不做导出 Excel**(除非跑哥点名要;原 Excel 本身就是要被替代的东西)。
- 不做历史逐日回填页(选到哪天算哪天即可)。
- 不动其它看板。

## 3. 数据源与口径

### 3.1 底表

- `trade_YYYYMM`(销售单头)+ `trade_goods_YYYYMM`(订单商品明细,`trade_id` 关联)。
- **发货日** = `trade.consign_time`(发货确认时间;trade 表本就按它分月)。取数用左闭右开区间 `consign_time >= 当日 AND < 次日`,当月累计用 `>= 当月1号 AND < 次日`。
- **只算 4 个仓**(`trade.warehouse_name IN`):
  - `南京委外成品仓-公司仓-委外`
  - `天津委外仓-公司仓-外仓`
  - `松鲜鲜&大地密码云仓`
  - `长沙委外成品仓-公司仓-外仓`
- **订单类型**:**只算销售 `trade_type = 1`**(跑哥定),排除售后发货(7)/退货(2)/售后退货(8)/仅退款(12)。

### 3.2 已验证(2026-07-01)

- 4 个仓在 `trade_202606` 均有真实发货数据(6/29 共 15515 单)。
- 货品组合口径已用真数据复现:按订单商品明细(货品×数量)组成"篮子签名"分组、按订单数排序,TOP 结果与 `销售日报.xlsx` 第三块逐条对上(松茸有机酱油100mL、新品100g+35g、85g、九谷猴菇山药面60g×8 等)。

### 3.3 已知不可靠

- `销售日报.xlsx` 是**结构模板**:当日半边全 `#DIV/0!`(没填),当月数是样例(单天真实值已 > 它的整月值)。**不以它的具体数字为准**,以逻辑为准,上线前拿真数据跟业务对一版校准。

## 4. 映射表落库

现在都在 Excel,一期落成 BI 表(建表 + 一次性导入 + 后续可维护)。

### 4.1 `dim_sales_channel_map`(渠道映射)

来源 `RPA映射表.xlsx / 销售渠道映射`(322 条)。**join 键 = `shop_name`**(实测我们 `shop_name` 与该表「销售渠道I」同格式:`ds-京东-店铺名` / `社媒-抖音-店铺名` / `sy-有赞-…` / `分销X组-…`)。

| 列 | 说明 |
|---|---|
| shop_name | 店铺(= 销售渠道I),join 键 |
| channel | 渠道(销售渠道II:京东/拼多多/天猫/唯品会/抖音/小红书/…) |
| platform | 平台(社媒/电商/其他),按下方分组表派生 |

**平台→渠道 分组**(照报表版式):

- 社媒:抖音、视频小店、小红书、快手
- 电商:拼多多、天猫、京东、唯品会
- 其他:分销、私域、线下、新零售、其它

> 待校准:映射表「渠道」列是 电商/私域/分销… 与报表「平台」分组不完全同名,导入时按上表把 渠道 归到 平台;无法归类的落「其它」。

### 4.2 `dim_goods_pack_spec`(箱规托规)

**join 键 = `goods_no`**。合并两个来源:

- 箱规打底:`RPA映射表.xlsx / 箱规映射`(1226 条,货品编号→箱规)。货品编号混编码:`S3-…`/`C4-…`/`sxx…` 直接是 goods_no;**纯数字是被 Excel 吃了前导 0 的 8 位码,导入时 `LPAD(code,8,'0')` 再对 goods_no**。
- 托规补充:`箱规拖规.xlsx`(52 条主力品,名称→箱规+托规)。**按 `名称` = `goods_name` 映射到 goods_no**(名称干净、实测可对)。

| 列 | 说明 |
|---|---|
| goods_no | 货品编码,join 键 |
| box_qty | 箱规 = 每箱单瓶数 |
| pallet_box_qty | 托规 = 每托箱数(仅主力 52 品有;其余为空,托数不出) |

- 箱数 = 单瓶 ÷ box_qty;托数 = 箱数 ÷ pallet_box_qty。
- 对不上箱规的少量 SKU:名称/条码兜底,仍对不上则该品箱数/托数留空、不报错。

### 4.3 赠品/组合

用现成 `goods_blend_detail`(8270 组合 BOM,含 `is_giveaway`)+ `goods.is_package_good`,不新建表。

## 5. 三块算法(当日 + 当月累计各算一遍)

统一先取"4 仓 + 发货日区间 + trade_type=1"的订单集 O 及其明细 L(join `dim_sales_channel_map`、`dim_goods_pack_spec`)。

### 5.1 平台/渠道汇总

按 platform / channel 分组:

- 发货单数 = `COUNT(DISTINCT trade_id)`
- 单瓶数 = `SUM(sell_count × box_qty)`(**按箱规展开成单瓶**,跑哥定;缺箱规的少量 SKU 暂 ×1)
- 重量 = `SUM(trade.estimate_weight)`(**用销售单头预估重量**,订单级;跑哥定)。**单位是克**,单均重量(kg) = Σweight ÷ 1000 ÷ 单数(实测 6/29 均值 1341g≈1.34kg,与报表 1.41kg 同量级)。覆盖 99.7%;`trade` 另有 `package_weight`(含包装,均值 1.5kg),默认 estimate_weight
- 单件比 = 单瓶 ÷ 单数;单均重量 = 重量 ÷ 单数
- 占比 = 该行单数 ÷ 总单数;环比 = 当日单数 ÷ 前一发货日单数 − 1
- 层级:平台合计行 = 旗下渠道求和(社媒/电商/其他)+ 总计行

### 5.2 TOP10 单品

按 `goods_no` 聚合:

- 发货单瓶数 = `SUM(sell_count × box_qty)`(按箱规展开),按它排序取前 10 + 合计行
- 箱数 = 单瓶 ÷ box_qty(= `SUM(sell_count)`);托数 = 箱数 ÷ pallet_box_qty
- 占比、环比同上

### 5.3 TOP10 货品组合

- 每个订单构造"篮子签名" = 该单所有明细行 `goods_no + 数量` 排序拼接
- 按签名分组,`COUNT(DISTINCT trade_id)` = 该组合订单数
- 按订单数排前 10 + 合计行;展示名用 `goods_name(数量)` 拼接(签名用**原始 sell_count 不展开**)
- 该组合的 单瓶/重量/单均 口径同渠道块(单瓶 = Σ sell_count×箱规、重量 = Σ 订单 estimate_weight)
- 注意 `group_concat_max_len` 要调大(默认 1024 会截断长篮子)

## 6. 接口

一个只读接口(仅超管/供应链权限):

```
GET /api/supply-chain/sales-daily-report?date=YYYY-MM-DD
```

返回三块结构化数据(channelRows / topGoods / topCombos),每块含当日 + 当月累计字段。后端新增 `handler/supply_chain_sales_daily_report.go`,复用 supply-chain 现有取数/鉴权套路。

- `date` 缺省 = 最近有发货数据的日期。
- 长查询加 timeout,避免锁链(见项目规约)。

## 7. 待校准口径(上线前拿真数据 + 业务确认)

1. ✅ **单瓶换算**(跑哥已定):单瓶 = `sell_count × 箱规`(按箱规展开)。缺箱规的少量 SKU 暂 ×1,建库时把箱规映射补齐。
2. ✅ **只算销售**(跑哥已定):`trade_type = 1`,不含售后发货(7)。
3. ✅ **重量**(跑哥已定):用销售单头 `trade.estimate_weight`(订单级)。仅剩小确认:estimate_weight vs package_weight 用哪个(默认 estimate_weight)。
4. **平台分组**:渠道→平台归类表最终以业务确认为准。
5. 上线前拿一个真实发货日,SQL 复核三块合计与页面一致(**不以模板 Excel 数字为准**)。

## 12. code-review 二审结论(2026-07-01, /code-review max)

已修 4 处 correctness(commit cf419ec):
